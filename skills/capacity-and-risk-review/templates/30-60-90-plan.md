# 30/60/90-Day Infrastructure Improvement Plan

Use this when a capacity/risk review finds issues that cannot all be fixed immediately. Keep actions evidence-based and tied to measurable outcomes.

## Guiding principles

- Prioritize no-regret actions that reduce outage, data-loss, security, or recovery risk.
- Separate urgent mitigations from structural improvements.
- Prefer actions with clear evidence, owner, rollback, and validation.
- Do not schedule disruptive work without an approved change window.

## Immediate / 0-7 days

Purpose: stabilize known risk and collect missing evidence.

| Action | Why now | Risk addressed | Owner | Evidence needed | Validation | Approval needed |
|---|---|---|---|---|---|---|
| Example: verify last successful restore test for critical backup set | Backup existence is not recovery evidence | restore/recovery |  | backup job logs, restore test record | test restore completed or gap documented | yes/no |

## 30 days

Purpose: remove high-probability/high-impact failure modes.

| Action | Target state | Dependency | Effort | Success metric | Rollback/exit criterion |
|---|---|---|---|---|---|
| Example: define SLO and alert for authentication service availability | alert maps to user impact | monitoring data source | medium | actionable alert with owner and runbook | disable noisy alert and revise threshold |
|  |  |  |  |  |  |

## 60 days

Purpose: improve resilience, observability, and operational repeatability.

| Action | Failure mode reduced | Evidence of completion | Operational handoff |
|---|---|---|---|
| Example: document and test firewall rollback procedure | bad rule blocks service | tested config backup + rollback note | runbook linked in change template |
|  |  |  |  |

## 90 days

Purpose: institutionalize the improvement and prevent regression.

| Action | Control/process | Review cadence | Metric |
|---|---|---|---|
| Example: quarterly restore drill for critical systems | recovery readiness | quarterly | RTO/RPO test pass rate |

## Review checklist

- [ ] Highest risks have owners and dates.
- [ ] Each disruptive action has a change plan.
- [ ] Each recommendation has a validation method.
- [ ] Missing evidence is called out explicitly.
- [ ] The plan distinguishes mitigation from permanent fix.
